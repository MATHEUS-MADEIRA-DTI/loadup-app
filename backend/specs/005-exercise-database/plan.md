# Implementation Plan: Exercise Database

**Branch**: `005-exercise-database` | **Date**: 2026-05-13 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/005-exercise-database/spec.md`

---

## Summary

This feature replaces the external API dependency (spec 003 API Ninjas) with a local static JSON database of 172 exercises in Portuguese (pt-BR). Search now queries local JSON instead of making network calls to external APIs, providing instant results (<100ms). Exercise data includes name, muscle group (expanded from 8 to 11 groups), YouTube video URLs, and optional tips. A new CSV import feature allows bulk adding of exercises to training days. The translation service (spec 005 previous version) is removed entirely—all data is pre-translated in the database.

---

**Scale/Scope**: Personal project; 172 exercises in local database; CSV import used by authenticated users/coaches

### Key Design Decisions

#### D001 — Load exercises.json at application startup

**Decision**: ExerciseDatabaseService loads and caches exercises.json on startup via OnModuleInit lifecycle hook. Searches query the in-memory array.

**Rationale**: Avoids file I/O on every search request; provides instant search performance; fail-fast if file is missing or invalid.

**Pattern**:

```typescript
import * as path from 'path';
import * as fs from 'fs';

@Injectable()
export class ExerciseDatabaseService implements OnModuleInit {
  private exercises: Exercise[] = [];

  async onModuleInit() {
    const filePath = path.join(__dirname, '..', '..', 'assets', 'exercises.json');
    this.exercises = JSON.parse(fs.readFileSync(filePath, 'utf-8'));
    this.logger.log(`Loaded ${this.exercises.length} exercises from database`);
  }

  search(name?: string, muscle?: string): Exercise[] {
    let results = this.exercises;
    if (name) {
      results = results.filter((ex) => ex.name.toLowerCase().includes(name.toLowerCase()));
    }
    if (muscle) {
      results = results.filter((ex) => ex.muscleGroup === muscle);
    }
    return results;
  }
}
```

#### D002 — Case-insensitive partial name matching

**Decision**: Search filters by substring match, case-insensitive. `"Supino"`, `"SUPINO"`, `"supino"` all match "Supino Reto Barra".

**Rationale**: User-friendly search; flexible query; matches UI search bar behavior.

#### D003 — CSV import validation with specific error reporting

**Decision**: Parse all rows using Node.js built-in string operations (split), collect validation errors with row numbers and field details. Return all errors at once (not fail-fast on first error). Series (tipo_serie, repeticoes) are NOT included in CSV—user configures series manually after import (same workflow as search).

**Rationale**: Users can see all problems in their CSV in one go; prevents repeated upload cycles. Built-in string operations eliminate external dependencies. Simplified format focuses on exercise metadata only (name, muscle group, video, tip). Series configuration is a separate user action, consistent with how searched exercises are added to training days.

**Pattern**:

```typescript
validateCsvRow(row: any, rowIndex: number): ValidationError[] {
  const errors: ValidationError[] = [];
  if (!row.nome || row.nome.trim() === '')
    errors.push({ row: rowIndex, field: 'nome', message: 'Required field' });
  if (!row.grupo_muscular || !Object.values(MuscleGroup).includes(row.grupo_muscular))
    errors.push({ row: rowIndex, field: 'grupo_muscular', message: `Invalid or missing group: ${row.grupo_muscular}` });
  if (row.video_url && !this.isValidUrl(row.video_url))
    errors.push({ row: rowIndex, field: 'video_url', message: 'Invalid URL format' });
  return errors;
}
```

#### D004 — No external API calls

**Decision**: All exercise data comes from local JSON file. No API Ninjas, no Lingva, no MyMemory. Search endpoint makes zero HTTP requests.

**Rationale**: Eliminates external dependencies, improves reliability, enables offline search, provides predictable performance.

#### D005 — Keep search response format identical to existing API

**Decision**: Response DTO structure remains unchanged from spec 003 (if frontend expects specific format). This ensures backward compatibility.

**Rationale**: Frontend code doesn't need to be updated; gradual migration possible if needed later.

#### D006 — CSV template includes comments

**Decision**: Generated CSV template includes header comment lines explaining required vs. optional fields, format rules, etc.

**Rationale**: Users don't need to read documentation; template is self-explanatory.

### Integration with Spec 001

The Exercise entity defined in spec 001 is extended with three fields:

- `videoUrl?: string` — YouTube URL for exercise execution (comes from database)
- `tip?: string` — Optional tip text or video URL
- `database: boolean` — Flag indicating exercise source (default true for database exercises)

### Integration with Spec 003

This feature **replaces** the exercises-api module behavior from spec 003:

- Old: External API call to API Ninjas → translation to PT-BR → cache in MongoDB
- New: Local JSON search → return immediately → no caching needed (data is always current)

The `/exercises/search` endpoint signature remains unchanged for frontend compatibility.

## Execution Flow

```
User searches: GET /exercises/search?name=supino&muscle=Peito
     ↓
ExercisesApiController receives request
     ↓
ExerciseDatabaseService.search('supino', 'Peito')
     ↓
Filter in-memory exercises.json array
     ↓
Return matching exercises with videoUrl, tip fields
     ↓
Frontend receives exercises, renders with video links if tip starts with "https://"
```

```
User imports CSV: POST /training-sheet/days/Monday/exercises/import
     ↓
TrainingSheetController receives multipart file
     ↓
CsvImportService.validateCsvFile(buffer)
     ↓
Parse CSV (nome;grupo_muscular;video_url;dica) → collect validation errors
     ↓
If errors: return 400 with error array (specific row/field details)
     ↓
If valid: TrainingSheetService.addExercisesFromCsv(dayOfWeek, csvRows)
     ↓
Create Exercise objects: { name, muscleGroup, videoUrl, tip, series: [], database: true }
     ↓
Add to training day exercises array with EMPTY series (user configures series manually)
     ↓
Return updated training day
     ↓
Frontend: User views imported exercises, adds series per exercise (same workflow as search)
```
